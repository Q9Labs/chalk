Implementation and qualification

Freeze once, build in three lanes, qualify the integrated release

Do not give the entire recorder to one agent. Give every lane the full recorder context, one companion spec as execution authority, its owned paths and phases, and a separate worktree from the same frozen baseline.

Background

The recorder is too broad for a single implementation branch: the control service, native media capture, graphics rendering, and real staging qualification need different tools and proof. They still share identities, job shapes, storage authority, events, and finalization.

The safe split is sequential at the boundaries and parallel inside implementation. First freeze the shared contracts in one baseline. Then build control, capture, and rendering independently. Merge and prove the local topology before qualification touches approved staging.

What exists today

Foundation already present

  • recording-facing SDK types and control routes;
  • reservation and admission foundations;
  • a direct Cloudflare room, track, and renegotiation proof;
  • transcription contracts and chunk jobs;
  • infrastructure, health, and gate foundations.

The product path is still missing

  • production worker identity and revocation;
  • the durable fenced job protocol;
  • the native selective capture loop;
  • encrypted bundle and timeline persistence;
  • the graphics executor and transactional finalization;
  • integrated staging evidence and cleanup proof.
A provider proof is not a recorder. The system remains not done until a real reservation becomes a verified video and transcript handoff under failure, cleanup, and deadline tests.

One execution authority per lane

Lane 1

Control plane

Owns worker identity, durable jobs, leases and fences, object and key authority, admission, finalization, reconciliation, observability, and the shared private route surface.

Authority: control-plane companion · phases C0–C6.

Lane 2

Cloudflare capture

Owns the native media client, selected encoded tracks, track identity, timeline, encrypted bundles, keyframes, gaps, provider recovery, and bounded machine behavior.

Authority: capture companion · phases K0–K6.

Lane 3

Render and finalization worker

Owns input verification, decryption, deterministic graphics, final video, speaker map, audio chunks, output verification, resource ceilings, and deadline evidence.

Authority: render companion · phases R0–R6.

Lane 4 · after integration

Staging qualification

Owns exact target confirmation, immutable deployment, provider and security proof, workload ceilings, failure drills, cleanup, observability, cost, binary verdict, and return to zero compute.

Authority: qualification companion · phases Q0–Q6.

What each implementation agent receives

All context

The umbrella recorder spec, infrastructure readiness, all four seam companions, transcription, and the frozen shared-contract artifacts.

One authority

Only the owned seam companion authorizes implementation. Other specs explain boundaries and consumers.

One worktree

A separate branch and worktree from the exact baseline commit, with explicit owned paths and no shared writing surface.

Exact phases

Named checklist phases, stopping point, exclusions, and which real outputs unlock the next lane.

Observed gates

Focused tests, local topology, failures, observability, cleanup, and the repository gate before handoff.

No invented choices

An open contract or staging binding stops the affected work; the agent cannot guess it from credentials or defaults.

Program gates

G0

Understand and ratify

Review this guide, accept or change each proposed shared default, then rewrite affected specs in place.

G1

Freeze and prove shared contracts

Commit versioned schemas, golden fixtures, identity, provider proof, assignments, object intents, finalization tests, and one immutable baseline.

G2

Implement three local lanes

Control C1–C5, capture K2–K5, and render R1–R5 pass in their owned worktrees.

G3

Integrate locally

One reservation becomes transcript jobs through restart, node loss, stale fence, duplicate report, cleanup, and database-backed recovery.

G4

Qualify the immutable release

Capture K6, render R6, and qualification Q0–Q5 pass against exact approved targets, quotas, cost, ceilings, and deadlines.

G5

Close safely

Record the binary verdict, delete temporary media, revoke temporary authority, prove zero recorder compute, and leave staging dormant.

Scope and stopping point

✓ In scope

  • safe admission through video and transcript handoff;
  • worker identity, job authority, encryption, storage, retries, and cleanup;
  • native Cloudflare capture and deterministic graphics rendering;
  • observability, health, security, cost, and real staging proof;
  • four lane boundaries and one worktree per writing owner.

× Deliberately out

  • production enablement or mutation;
  • browser, client-side, or managed RealtimeKit recording;
  • arbitrary post-meeting gallery edits or voice identity;
  • rooms over ten people or recordings over two hours;
  • one agent implementing the whole program from this guide.
This orientation guide stops after Hasan understands the system and the shared choices are explicit. Contract implementation begins only after G0; staging begins only after G3 and explicit staging approval.
