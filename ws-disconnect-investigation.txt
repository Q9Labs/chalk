5. CODE ANALYSIS: WEBSOCKET CONNECTION FLOW
--------------------------------------------------------------------------------
File: apps/api/internal/interfaces/http/handlers/websocket.go

1. Client sends JWT token via Sec-WebSocket-Protocol header
2. Handler validates JWT and extracts room_id, participant_id, tenant_id
3. websocket.Accept() upgrades HTTP to WebSocket
4. NewClient() creates client struct with hub reference
5. hub.Register(client) sends client to registration channel
6. client.Start(ctx) launches read/write goroutines
7. client.Wait() blocks until done channel is closed

POTENTIAL ISSUE on line 144-145:
  ctx, cancel := context.WithCancel(c.Request.Context())
  defer cancel()

The context is derived from the HTTP request context (c.Request.Context()).
If the request context is canceled (e.g., ALB timeout, API Gateway timeout),
the WebSocket context will also be canceled, causing immediate disconnect.

8. ROOT CAUSE HYPOTHESES (Ranked by likelihood)
--------------------------------------------------------------------------------

HYPOTHESIS 1: API Gateway / ALB WebSocket Timeout (HIGH PROBABILITY)
  The API Gateway or ALB has idle timeout settings that may be closing the
  WebSocket connection. Default ALB idle timeout is 60 seconds, but the
  connection dropped in <1 second.

  Check: API Gateway WebSocket throttling settings:
    - websocket_throttling_burst_limit = 1000
    - websocket_throttling_rate_limit = 2000

  The WebSocket API log group has 0 bytes stored, suggesting WS connections
  may not be routing through API Gateway correctly.

HYPOTHESIS 2: Cloudflare WebSocket Timeout (MEDIUM PROBABILITY)
  Cloudflare has its own WebSocket timeout (typically 100 seconds idle).
  However, connection failed in <1 second, which rules out idle timeout.
  Could be:
    - Rate limiting at Cloudflare level
    - WebSocket upgrade blocked

HYPOTHESIS 3: Context Cancellation Race (MEDIUM PROBABILITY)
  The WebSocket handler uses c.Request.Context() which may be canceled
  if the original HTTP request times out or the client disconnects during
  upgrade.

HYPOTHESIS 4: Hub Registration Race Condition (LOW PROBABILITY)
  The hub.Register() is asynchronous (channel-based). If the client
  attempts to receive messages before registration completes, it may fail.
  However, the code handles this with buffered channels.

HYPOTHESIS 5: Memory/Resource Exhaustion (LOW PROBABILITY)
  ECS task is small (256 CPU, 512MB). Under load, goroutine creation
  for WebSocket pumps could fail. Check Container Insights for memory usage.
